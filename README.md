**Sample: idempotent user creation in `provision.sh`**
```bash
create_user() {
  local user=$1
  if id "$user" &>/dev/null; then
    log "SKIP" "User '$user' already exists"
  else
    useradd -m -s /bin/bash "$user" && log "OK" "Created user '$user'"
  fi
}
```

### [`unsloth-zoo`](https://github.com/unslothai/unsloth-zoo) — GGUF Shard Validation Fix

**Contribution date:** Early 2026 | **Type:** Bug fix / validation improvement

**Problem I found:**  
When exporting multi-shard GGUF models, the existing validation was too permissive — it did not catch missing shard files, wrong shard counts, or sequences with gaps. These errors only surfaced late in the export, wasting compute time.

**What the fix does:**
- Detects missing shard files before export completes
- Validates that the declared shard count matches actual file count
- Catches invalid or non-sequential shard ordering
- Fails early and clearly instead of silently producing corrupt output

```python
# Before: basic existence check only
if not os.path.exists(shard_path):
    raise FileNotFoundError(...)

# After: full shard sequence validation
def validate_gguf_shards(shard_paths: list[str], expected_count: int):
    if len(shard_paths) != expected_count:
        raise ValueError(
            f"Expected {expected_count} shards, found {len(shard_paths)}"
        )
    for i, path in enumerate(sorted(shard_paths)):
        if not os.path.exists(path):
            raise FileNotFoundError(f"Missing shard {i}: {path}")
    # sequence gap detection
    indices = [extract_shard_index(p) for p in shard_paths]
    if indices != list(range(expected_count)):
        raise ValueError(f"Invalid shard sequence: {indices}")
```

**Why it matters:** Fine-tuning and quantization pipelines (LoRA → GGUF export) break silently without this. The fix makes the failure loud and early.
